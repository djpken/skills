# Codebase Design（程式碼庫設計）

設計**深層模組**：一個小介面背後隱藏大量行為，放置在乾淨的接縫處，可透過該介面測試。在設計或重構程式碼的任何地方都使用這套語言和原則。目標是給呼叫者槓桿、給維護者局部性、給所有人可測試性。

## 術語表

請精確使用這些術語——不要用「component」、「service」、「API」或「boundary」替代。一致的語言才是重點。

**模組（Module）** — 任何有介面和實作的東西。刻意不限規模：可以是一個函式、類別、套件或跨層切片。_避免_：unit、component、service。

**介面（Interface）** — 呼叫者必須知道才能正確使用模組的一切：型別簽名，也包括不變性、排序約束、錯誤模式、必要設定和效能特性。_避免_：API、signature（太窄——它們只指型別層面的表面）。

**實作（Implementation）** — 模組內部的東西，其程式碼主體。與**轉接器（Adapter）**不同：一個東西可以是小轉接器加大實作（Postgres repo），也可以是大轉接器加小實作（記憶體假物件）。接縫是討論主題時用「adapter」；否則用「implementation」。

**深度（Depth）** — 介面的槓桿：呼叫者（或測試）每單位學習成本可以驅動的行為量。模組**深**是指大量行為躲在小介面背後，**淺**是指介面幾乎和實作一樣複雜。

**接縫（Seam）** _（Michael Feathers）_ — 一個可以在不編輯該處就改變行為的地方；模組介面所在的*位置*。接縫放在哪裡本身就是個設計決策，與背後放什麼無關。_避免_：boundary（與 DDD 的 bounded context 混用）。

**轉接器（Adapter）** — 在接縫處滿足介面的具體東西。描述的是*角色*（填補哪個槽位），而不是本質（內部是什麼）。

**槓桿（Leverage）** — 呼叫者從深度獲得的東西：每單位學習到的介面能取得更多能力。一個實作回報給 N 個呼叫點和 M 個測試。

**局部性（Locality）** — 維護者從深度獲得的東西：變更、bug、知識和驗證集中在一個地方，而不是分散到各個呼叫者。修一次，處處修好。

## 深 vs 淺

**深層模組** = 小介面 + 大量實作：

```
┌─────────────────────┐
│   Small Interface   │  ← 少量方法，簡單參數
├─────────────────────┤
│                     │
│  Deep Implementation│  ← 複雜邏輯隱藏其中
│                     │
└─────────────────────┘
```

**淺層模組** = 大介面 + 少量實作（避免）：

```
┌─────────────────────────────────┐
│       Large Interface           │  ← 大量方法，複雜參數
├─────────────────────────────────┤
│  Thin Implementation            │  ← 只是透傳
└─────────────────────────────────┘
```

設計介面時，問自己：

- 我能減少方法數量嗎？
- 我能簡化參數嗎？
- 我能把更多複雜度隱藏在裡面嗎？

## 原則

- **深度是介面的屬性，不是實作的屬性。** 一個深層模組的內部可以由小型、可模擬、可替換的部分組成——它們只是不屬於介面的一部分。一個模組可以有**內部接縫**（對其實作私有，供自己的測試使用），也有**外部接縫**（在其介面處）。
- **刪除測試。** 想像刪除這個模組。如果複雜度消失了，它只是個透傳。如果複雜度在 N 個呼叫者身上重現，它是在賺取它的存在。
- **介面是測試表面。** 呼叫者和測試穿越同一個接縫。如果你想*繞過*介面測試，這個模組的形狀可能是錯的。
- **一個轉接器意味著假設接縫。兩個轉接器意味著真實接縫。** 除非真的有東西跨越它而變化，否則不要引入接縫。

## 為可測試性設計

好的介面讓測試變得自然：

1. **接受依賴，不要建立依賴。**

   ```typescript
   // 可測試
   function processOrder(order, paymentGateway) {}

   // 難以測試
   function processOrder(order) {
     const gateway = new StripeGateway();
   }
   ```

2. **回傳結果，不要產生副作用。**

   ```typescript
   // 可測試
   function calculateDiscount(cart): Discount {}

   // 難以測試
   function applyDiscount(cart): void {
     cart.total -= discount;
   }
   ```

3. **小介面表面積。** 方法越少 = 需要的測試越少。參數越少 = 測試設定越簡單。

## 關係

- 一個**模組**恰好有一個**介面**（它呈現給呼叫者和測試的表面）。
- **深度**是**模組**的屬性，以其**介面**為基準衡量。
- **接縫**是**模組**的**介面**所在的地方。
- **轉接器**坐落在**接縫**處並滿足**介面**。
- **深度**為呼叫者產生**槓桿**，為維護者產生**局部性**。

## 被拒絕的框架

- **深度作為實作行數對介面行數的比率**（Ousterhout）：獎勵填充實作。我們改用深度即槓桿。
- **「介面」作為 TypeScript `interface` 關鍵字或類別的公開方法**：太窄——這裡的介面包括呼叫者必須知道的每一個事實。
- **「Boundary」**：與 DDD 的 bounded context 混用。說**接縫（seam）**或**介面（interface）**。
