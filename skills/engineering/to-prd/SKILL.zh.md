這個 skill 取得目前對話 context 和程式碼庫理解，並產生 PRD。**不要**訪談使用者——只是綜合你已知道的東西。

Issue 追蹤器和 triage 標籤詞彙應該已提供給你——如果沒有，執行 `/setup-matt-pocock-skills`。

## 流程

1. 如果你還沒有，探索 repo 以了解程式碼庫的目前狀態。在整個 PRD 中使用專案的領域術語表詞彙，並遵循你正在觸碰的區域的任何 ADR。

2. 勾勒出你要測試功能的接縫。應該優先使用現有接縫而不是新的。盡可能使用最高的接縫。如果需要新接縫，在你能做到的最高點提議它們。程式碼庫中的接縫越少越好——理想數量是一個。

   與使用者確認這些接縫符合他們的期望。

3. 使用下面的模板寫 PRD，然後將其發布到專案 issue 追蹤器。應用 `ready-for-agent` triage 標籤——不需要額外的 triage。

<prd-template>

## 問題陳述

使用者面臨的問題，從使用者的角度。

## 解決方案

問題的解決方案，從使用者的角度。

## 使用者故事

一個**長的**、編號的使用者故事列表。每個使用者故事應該採用以下格式：

1. 作為一個 <角色>，我想要一個 <功能>，以便 <好處>

<user-story-example>
1. 作為一個行動銀行客戶，我想看到我帳戶的餘額，以便我能對我的消費做出更明智的決定
</user-story-example>

這個使用者故事列表應該非常廣泛，涵蓋功能的各個方面。

## 實作決策

已做出的實作決策列表。這可以包括：

- 將要建立/修改的模組
- 將要修改的那些模組的介面
- 來自開發者的技術澄清
- 架構決策
- Schema 變更
- API 合約
- 具體互動

不要包括具體的檔案路徑或程式碼片段。它們可能很快就會過時。

例外：如果原型產生了一個比散文更精確地編碼決策的片段（狀態機、reducer、schema、型別形狀），在相關決策中內嵌它，並簡單注明它來自原型。修剪到決策豐富的部分——不是一個可運作的演示，只是重要的位元。

## 測試決策

已做出的測試決策列表。包括：

- 什麼是好測試的描述（只測試外部行為，而不是實作細節）
- 哪些模組將被測試
- 測試的先例（即程式碼庫中類似類型的測試）

## 範圍外

對此 PRD 範圍外事項的描述。

## 進一步說明

關於功能的任何進一步說明。

</prd-template>
