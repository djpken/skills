# To Issues（轉為 Issues）

使用垂直切片（曳光彈）將計畫拆解成可獨立認領的 issues。

Issue 追蹤器和 triage 標籤詞彙應該已提供給你——如果沒有，執行 `/setup-matt-pocock-skills`。

## 流程

### 1. 收集背景

從對話 context 中已有的內容開始工作。如果使用者傳遞了 issue 參考（issue 號碼、URL 或路徑）作為參數，從 issue 追蹤器取得它並讀取其完整正文和評論。

### 2. 探索程式碼庫（可選）

如果你還沒有探索程式碼庫，這樣做以了解程式碼的目前狀態。Issue 標題和描述應該使用專案的領域術語表詞彙，並遵循你正在觸碰的區域的 ADR。

尋找預重構程式碼的機會，讓實作更容易。「先讓變更變容易，再讓容易的變更發生。」

### 3. 起草垂直切片

將計畫拆解成**曳光彈** issues。每個 issue 是一個薄的垂直切片，端對端切穿所有整合層，而不是一層的水平切片。

<vertical-slice-rules>

- 每個切片提供一條穿越每個層（schema、API、UI、測試）的窄但完整的路徑
- 完成的切片可以獨立展示或驗證
- 任何預重構都應該先完成

</vertical-slice-rules>

### 4. 測驗使用者

將提議的分解以編號列表呈現。對每個切片顯示：

- **標題**：簡短的描述性名稱
- **被阻擋**：哪些其他切片（如果有的話）必須先完成
- **涵蓋的使用者故事**：這個切片解決哪些使用者故事（如果來源材料有的話）

詢問使用者：

- 粒度感覺對嗎？（太粗 / 太細）
- 依賴關係正確嗎？
- 任何切片應該合併或進一步拆分嗎？

反覆迭代直到使用者批准分解。

### 5. 將 issues 發布到 issue 追蹤器

對每個批准的切片，向 issue 追蹤器發布一個新 issue。使用下面的 issue 正文模板。這些 issues 被認為已準備好給 AFK agent，所以用正確的 triage 標籤發布它們，除非另有指示。

按依賴順序發布 issues（阻擋者先），這樣你就可以在「被阻擋」欄位中引用真實的 issue 識別符。

<issue-template>
## 父 Issue

在 issue 追蹤器上對父 issue 的引用（如果來源是現有 issue，否則省略此部分）。

## 要建構什麼

這個垂直切片的簡潔描述。描述端對端的行為，而不是逐層的實作。

避免具體的檔案路徑或程式碼片段——它們很快就會過時。例外：如果原型產生了一個比散文更精確地編碼決策的片段（狀態機、reducer、schema、型別形狀），在此內嵌並簡單注明它來自原型。修剪到決策豐富的部分——不是一個可運作的演示，只是重要的位元。

- [ ] 標準 2
- [ ] 標準 3

## 被阻擋

- 對阻擋 ticket 的引用（如果有的話）

或「無——可以立即開始」如果沒有阻擋者。

</issue-template>

不要關閉或修改任何父 issue。
