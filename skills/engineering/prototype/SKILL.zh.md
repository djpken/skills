# Prototype（原型）

原型是**回答一個問題的可拋棄程式碼**。問題決定了形狀。

## 選擇分支

確定正在回答哪個問題——從使用者的提示、周圍的程式碼，或在使用者在線時詢問：

- **「這個邏輯 / 狀態模型感覺對嗎？」** → [LOGIC.md](LOGIC.md)。建立一個小型互動式終端 app，將狀態機推過那些在紙上難以推理的案例。
- **「這應該長什麼樣子？」** → [UI.md](UI.md)。在一個 route 上生成幾個截然不同的 UI 變體，可透過 URL 搜尋參數和浮動底部欄切換。

這兩個分支產生非常不同的成品——搞錯這個就浪費了整個原型。如果問題真的模稜兩可且使用者無法聯繫，預設選擇更符合周圍程式碼的分支（後端模組 → 邏輯；頁面或元件 → UI），並在原型頂部說明這個假設。

## 適用於兩者的規則

1. **從第一天起就是可拋棄的，並清楚標記。** 將原型程式碼放在它實際會被使用的地方附近（在它正在原型化的模組或頁面旁邊），讓背景清楚——但要命名讓隨意的讀者能看出它是原型，而不是生產程式碼。對於可拋棄的 UI route，遵從專案已使用的任何路由慣例；不要發明新的頂層結構。
2. **一個指令可以執行。** 無論專案現有的任務執行器支援什麼——`pnpm <name>`、`python <path>`、`bun <path>` 等。使用者必須能在不思考的情況下啟動它。
3. **預設不持久化。** 狀態存在於記憶體中。持久化是原型*要檢查的東西*，不是它應該依賴的東西。如果問題明確涉及資料庫，使用一個草稿 DB 或一個帶有清晰「PROTOTYPE — 清除我」名稱的本地檔案。
4. **跳過精雕細琢。** 沒有測試、沒有錯誤處理（超過讓原型*可執行*的範圍）、沒有抽象。重點是快速學到東西然後刪除它。
5. **浮出狀態。** 每次動作後（邏輯）或每次變體切換時（UI），列印或渲染完整相關的狀態，讓使用者能看到什麼改變了。
6. **完成時刪除或吸收。** 當原型回答了它的問題，要麼刪除它，要麼將驗證過的決策折疊進真正的程式碼——不要讓它在 repo 中腐爛。

## 完成時

原型只有*答案*值得保留。在某個持久的地方捕獲它（commit 訊息、ADR、issue，或原型旁邊的 `NOTES.md`），連同它所回答的問題。如果使用者在線，那個捕獲是一次快速對話；如果不在，留下佔位符讓他們（或你，在下一次）在刪除原型之前填寫判決。
