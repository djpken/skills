# Diagnosing Bugs（診斷 Bug）

處理難搞 bug 的紀律。只有在明確有理由時才跳過階段。

探索程式碼庫時，讀取 `CONTEXT.md`（如果存在）以建立相關模組的清晰心智模型，並查看你正在觸碰的區域的 ADR。

## 第一階段 — 建立反饋迴路

**這才是技巧所在。** 其他的都是機械性工作。如果你對 bug 有一個**緊密的**通過/失敗訊號——一個在*這個* bug 上會變紅的訊號——你就能找到原因；二分法、假設測試和工具插樁都只是消耗它。如果你沒有，再多盯著程式碼看也救不了你。

在這裡投入不成比例的努力。**要積極。要有創意。拒絕放棄。**

### 建立的方式——大致依此順序嘗試

1. **失敗的測試**：在任何能觸達 bug 的接縫處——單元、整合、e2e。
2. **Curl / HTTP 腳本**：針對運行中的 dev server。
3. **CLI 呼叫**：使用 fixture 輸入，將 stdout 與已知正確的快照做 diff。
4. **Headless 瀏覽器腳本**（Playwright / Puppeteer）——驅動 UI，在 DOM/console/network 上斷言。
5. **重放捕獲的追蹤。** 將真實的網路請求 / payload / 事件日誌存到磁碟；在隔離環境中重放它穿越程式碼路徑。
6. **拋棄式測試框架。** 啟動系統的最小子集（一個服務，模擬依賴），用單一函式呼叫來驅動 bug 程式碼路徑。
7. **屬性 / 模糊測試迴路。** 如果 bug 是「有時輸出錯誤」，執行 1000 個隨機輸入，尋找失敗模式。
8. **二分法框架。** 如果 bug 出現在兩個已知狀態（commit、資料集、版本）之間，自動化「在狀態 X 啟動，檢查，重複」，讓你可以 `git bisect run` 它。
9. **差異迴路。** 用舊版 vs 新版（或兩個設定）執行相同輸入並做 diff。
10. **HITL bash 腳本。** 最後手段。如果需要人類點擊，用 `scripts/hitl-loop.template.sh` 驅動*他們*，讓迴路仍然有結構。捕獲的輸出回饋給你。

建立正確的反饋迴路，bug 就修了 90%。

### 收緊迴路

把迴路當產品對待。一旦你有了*一個*迴路，就**收緊**它：

- 我能讓它更快嗎？（快取設定、跳過不相關的初始化、縮小測試範圍。）
- 我能讓訊號更清晰嗎？（斷言具體症狀，而不是「沒有崩潰」。）
- 我能讓它更具確定性嗎？（固定時間、設定 RNG 種子、隔離檔案系統、凍結網路。）

一個 30 秒的不穩定迴路幾乎和沒有迴路一樣差；一個 2 秒的確定性迴路是緊密的——除錯的超能力。

### 非確定性 bug

目標不是乾淨的重現，而是**更高的重現率**。迴路觸發 100 次、並行化、增加壓力、縮小時序視窗、注入睡眠。50% 的偶發 bug 是可除錯的；1% 不是——持續提高率直到可除錯。

### 當你真的無法建立迴路時

停下來，明確說出來。列出你嘗試過的東西。請求使用者提供：(a) 能重現問題的環境存取、(b) 捕獲的成品（HAR 檔案、日誌轉儲、核心轉儲、帶時間戳的螢幕錄影）或 (c) 添加臨時生產工具插樁的許可。在沒有迴路的情況下**不要**繼續假設。

### 完成標準——一個變紅的緊密迴路

當迴路**緊密**且**能變紅**時，第一階段才算完成：你能說出**一個指令**——一個腳本路徑、一個測試呼叫、一個 curl——你**已經至少執行過一次**（貼上呼叫及其輸出），並且：

- [ ] **能變紅** — 它驅動實際的 bug 程式碼路徑並斷言**使用者確切的症狀**，讓它能在這個 bug 上變紅，修好後變綠。不是「執行時不報錯」——它必須能*抓住這個特定 bug*。
- [ ] **確定性** — 每次執行結果相同（不穩定 bug：每次高重現率，如上所述）。
- [ ] **快速** — 幾秒，不是幾分鐘。
- [ ] **Agent 可執行** — 你可以無人值守地執行它；只有透過 `scripts/hitl-loop.template.sh` 才需要人類。

如果你發現自己在這個指令存在之前就在閱讀程式碼來建立理論，**停止——直接跳到假設正是這個 skill 要防止的失敗。** 沒有能變紅的指令，就沒有第二階段。

## 第二階段 — 重現 + 最小化

執行迴路。看它變紅——bug 出現了。

確認：

- [ ] 迴路產生了**使用者**描述的失敗模式——而不是附近碰巧出現的不同失敗。錯誤的 bug = 錯誤的修復。
- [ ] 失敗在多次執行中可重現（或對非確定性 bug，重現率足夠高以供除錯）。
- [ ] 你已捕獲確切的症狀（錯誤訊息、錯誤輸出、慢時序），以便後續階段能驗證修復確實解決了它。

### 最小化

一旦它變紅，將重現縮小到**仍然變紅的最小場景**。逐一削減輸入、呼叫者、設定、資料和步驟，每次削減後重新執行迴路——只保留對失敗有支撐作用的東西。

為什麼要費心：最小重現在第三階段縮小假設空間（剩下更少要懷疑的活動部件），並成為第五階段的乾淨回歸測試。

當**每個剩餘元素都有支撐作用**時完成——移除其中任何一個都會讓迴路變綠。

在重現**且**最小化之前不要繼續。

## 第三階段 — 假設

在測試任何假設之前，生成 **3–5 個已排名的假設**。單一假設的生成會錨定在第一個似乎合理的想法上。

每個假設都必須是**可偽證的**：說明它所做的預測。

> 格式：「如果 <X> 是原因，那麼 <改變 Y> 將使 bug 消失 / <改變 Z> 將使它更嚴重。」

如果你無法說明預測，這個假設只是直覺——丟棄或磨練它。

**在測試之前向使用者展示排名列表。** 他們通常有領域知識能立即重新排名（「我們剛部署了對 #3 的更改」），或知道他們已經排除的假設。便宜的檢查點，節省大量時間。如果使用者不在線，不要等——按你的排名繼續。

## 第四階段 — 工具插樁

每個探針都必須映射到第三階段的具體預測。**每次只改變一個變量。**

工具偏好：

1. **除錯器 / REPL 檢查**（如果環境支援的話）。一個斷點勝過十個日誌。
2. **在區分假設的邊界處放目標日誌。**
3. 絕不「記錄所有東西然後 grep」。

**用唯一前綴標記每個除錯日誌**，例如 `[DEBUG-a4f2]`。最後清理變成一個單一的 grep。未標記的日誌存活；已標記的日誌消亡。

**效能分支。** 對效能退化，日誌通常是錯的。改為：建立基準測量（計時框架、`performance.now()`、剖析器、查詢計畫），然後進行二分法。先測量，再修復。

## 第五階段 — 修復 + 回歸測試

在修復之前寫回歸測試——但只在有**正確接縫**的情況下。

正確的接縫是測試在呼叫點以實際發生的方式驅動**真實 bug 模式**的地方。如果唯一可用的接縫太淺（當 bug 需要多個呼叫者時的單呼叫者測試、無法複製觸發 bug 的鏈的單元測試），那個接縫的回歸測試會給出錯誤的信心。

**如果不存在正確的接縫，這本身就是發現。** 記錄它。程式碼庫架構阻止了 bug 被鎖定。為下一階段標記此問題。

如果存在正確的接縫：

1. 將最小重現變成在那個接縫的失敗測試。
2. 看它失敗。
3. 應用修復。
4. 看它通過。
5. 對原始（未最小化的）場景重新執行第一階段反饋迴路。

## 第六階段 — 清理 + 事後回顧

宣告完成前必須的：

- [ ] 原始重現不再重現（重新執行第一階段迴路）
- [ ] 回歸測試通過（或無接縫已記錄）
- [ ] 所有 `[DEBUG-...]` 工具插樁已移除（`grep` 前綴）
- [ ] 拋棄式原型已刪除（或移至明確標記的除錯位置）
- [ ] 結果正確的假設已在 commit / PR 訊息中說明——讓下一個除錯者學習

**然後問：是什麼會防止這個 bug？** 如果答案涉及架構更改（沒有好的測試接縫、糾纏的呼叫者、隱藏的耦合），帶著具體情況移交給 `/improve-codebase-architecture` skill。在修復完成*後*提出建議，而不是之前——你現在擁有的資訊比你開始時更多。
