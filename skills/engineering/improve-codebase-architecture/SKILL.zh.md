# Improve Codebase Architecture（改善程式碼庫架構）

浮出架構摩擦點，並提出**深化機會**——將淺層模組轉化為深層模組的重構。目標是可測試性和 AI 可導覽性。

這個指令*受*專案領域模型的指引，建立在共享設計詞彙上：

- 執行 `/codebase-design` skill 取得架構詞彙（**模組、介面、深度、接縫、轉接器、槓桿、局部性**）及其原則（刪除測試、「介面是測試表面」、「一個轉接器 = 假設接縫，兩個 = 真實接縫」）。在每個建議中精確使用這些術語——不要漂移到「component」、「service」、「API」或「boundary」。
- `CONTEXT.md` 中的領域語言給好的接縫命名；`docs/adr/` 中的 ADR 記錄這個指令不應重新辯論的決策。

## 流程

### 1. 探索

首先讀取專案的領域術語表（`CONTEXT.md`）和你正在觸碰的區域的任何 ADR。

然後使用帶有 `subagent_type=Explore` 的 Agent 工具瀏覽程式碼庫。不要遵循僵硬的啟發式規則——有機地探索並記錄你體驗到摩擦的地方：

- 哪裡需要在許多小模組之間跳轉才能理解一個概念？
- 哪裡的模組是**淺**的——介面幾乎和實作一樣複雜？
- 哪裡的純函式只是為了可測試性而被提取出來，但真正的 bug 隱藏在它們的呼叫方式中（沒有**局部性**）？
- 哪裡的緊耦合模組在其接縫處洩漏？
- 程式碼庫的哪些部分未經測試，或難以通過其目前介面進行測試？

對你懷疑是淺層的任何東西應用**刪除測試**：刪除它會集中複雜度，還是只是移動它？「是，集中」是你想要的訊號。

### 2. 以 HTML 報告呈現候選項

將一個獨立的 HTML 檔案寫到 OS 臨時目錄，讓任何東西都不會落在 repo 中。從 `$TMPDIR` 解析臨時目錄，回退到 `/tmp`（Windows 上為 `%TEMP%`），並寫到 `<tmpdir>/architecture-review-<timestamp>.html`，讓每次執行都能得到新鮮的檔案。為使用者開啟它——Linux 上用 `xdg-open <path>`，macOS 上用 `open <path>`，Windows 上用 `start <path>`——並告訴他們絕對路徑。

報告使用 **Tailwind 透過 CDN** 進行佈局和樣式設定，使用 **Mermaid 透過 CDN** 在圖表可靠傳達結構時使用。混合 Mermaid 和手工製作的 CSS/SVG 視覺效果——當關係是圖形形狀時使用 Mermaid（呼叫圖、依賴關係、序列），當你想要更有編輯感的東西時使用手建的 div/SVG（質量圖、橫截面、折疊動畫）。每個候選項都有一個**前後視覺化**。要有視覺效果。

對每個候選項，渲染一張卡片，包含：

- **涉及的檔案** — 哪些檔案/模組涉及
- **問題** — 為什麼目前架構造成摩擦
- **解決方案** — 將要改變什麼的簡單英文描述
- **好處** — 用局部性和槓桿術語解釋，以及測試將如何改善
- **前後圖** — 並排，自定義繪製，說明淺層性和深化
- **建議強度** — `Strong`（強烈）、`Worth exploring`（值得探索）、`Speculative`（推測），渲染為徽章

報告末尾附**最佳建議**部分：你會先處理哪個候選項以及原因。

**為領域使用 CONTEXT.md 詞彙，為架構使用 `/codebase-design` 詞彙。** 如果 `CONTEXT.md` 定義了「Order」，就談「Order intake module」——而不是「FooBarHandler」，也不是「Order service」。

**ADR 衝突**：如果候選項與現有 ADR 相矛盾，只在摩擦真實到足以值得重新審視 ADR 時才浮出它。在卡片中清楚標記（例如，警告標注：_「與 ADR-0007 矛盾——但值得重新開啟，因為……」_）。不要列出 ADR 禁止的每個理論上的重構。

參見 [HTML-REPORT.md](HTML-REPORT.md) 了解完整的 HTML 腳手架、圖表模式和樣式指引。

還不要提出介面。寫完檔案後，問使用者：「你想探索哪些？」

### 3. 深度訪談迴路

一旦使用者選擇了候選項，執行 `/grilling` skill 與他們一起走過設計樹——約束、依賴、深化模組的形狀、接縫背後有什麼、哪些測試存活。

在決策結晶時即時發生副作用——執行 `/domain-modeling` skill 讓領域模型在進行中保持最新：

- **以 `CONTEXT.md` 中沒有的概念命名深化模組？** 把術語加到 `CONTEXT.md`。如果不存在，懶惰地建立檔案。
- **在對話中磨練模糊術語？** 立刻更新 `CONTEXT.md`。
- **使用者以有支撐的理由拒絕候選項？** 提供 ADR，框架為：_「你想讓我把這記錄為 ADR，讓未來的架構審查不會再次建議它嗎？」_ 只在理由真的會被未來探索者需要以避免再次建議相同東西時才提供——跳過短暫的理由（「現在不值得」）和顯而易見的理由。
- **想探索深化模組的替代介面？** 執行 `/codebase-design` skill 並使用其設計兩次並行子 agent 模式。
