# Test-Driven Development（測試驅動開發）

TDD 就是紅 → 綠迴路。這個 skill 是讓這個迴路產出值得保留的測試的參考：什麼是好測試、測試該寫在哪裡、反模式，以及迴路的規則。每個章節在每次循環都適用——在迴路之前和期間都要參考它們，而不是事後才看。

探索程式碼庫時，讀取 `CONTEXT.md`（如果存在），讓測試名稱和介面詞彙符合專案的領域語言，並遵循你正在觸碰的區域的 ADR。

## 什麼是好測試

測試應該透過公開介面驗證行為，而不是實作細節。程式碼可以完全改變；測試不應該。好的測試讀起來像規格——「使用者可以用有效購物車結帳」精確告訴你存在什麼能力——並且在重構後仍然存活，因為它不關心內部結構。

參見 [tests.md](tests.md) 了解範例，參見 [mocking.md](mocking.md) 了解模擬指引。

## 接縫——測試該寫在哪裡

**接縫（seam）** 是你測試的公開邊界：可以在不深入內部的情況下觀察行為的介面。測試永遠寫在接縫上，不對內部寫。

**只在事先商定的接縫上測試。** 寫任何測試之前，先寫下要測試的接縫並與使用者確認。不在未經確認的接縫上寫測試。你無法測試所有東西——事先商定接縫，才能讓測試精力落在關鍵路徑和複雜邏輯上，而不是每個邊界案例。

問：「公開介面應該長什麼樣子？哪些接縫該測試？」

## 反模式

- **與實作耦合** — 模擬內部協作者、測試私有方法，或透過側路驗證（例如直接查詢資料庫而不是使用介面）。特徵：你重構時測試失敗，但行為沒有改變。
- **套套邏輯（tautological）** — 斷言用程式碼本身的算法重新計算預期值（`expect(add(a, b)).toBe(a + b)`、用同樣方式手算的快照、常數斷言等於自己），因此它必然通過，永遠不可能與程式碼有分歧。預期值必須來自獨立的真相來源——已知正確的字面值、一個範例，或規格。
- **水平切片** — 先寫完所有測試，再寫所有實作。大量測試驗證的是*想像中的*行為：你測試的是事物的*形狀*而不是面向使用者的行為，測試對真實變化變得不敏感，而且你在理解實作之前就先定死了測試結構。改用**垂直切片**：一個測試 → 一個實作 → 重複，每個測試都是回應上一次循環所教會你的**曳光彈**。

## 迴路的規則

- **先紅後綠。** 先寫失敗的測試，再寫剛好足以通過它的程式碼。不要預測未來的測試，也不要加入推測性功能。
- **一次一個切片。** 每個循環一個接縫、一個測試、一個最小實作。
- **重構不是迴路的一部分。** 它屬於審查階段（見 `review` skill），不屬於紅 → 綠的實作循環。
