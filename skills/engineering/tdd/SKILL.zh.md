# Test-Driven Development（測試驅動開發）

## 哲學

**核心原則**：測試應該透過公開介面驗證行為，而不是實作細節。程式碼可以完全改變；測試不應該。

**好的測試**是整合風格的：它們透過公開 API 驅動真實的程式碼路徑。它們描述系統*做什麼*，而不是*怎麼做*。好的測試讀起來像規格——「使用者可以用有效購物車結帳」精確告訴你存在什麼能力。這些測試在重構後仍然存活，因為它們不關心內部結構。

**壞的測試**與實作耦合。它們模擬內部協作者、測試私有方法，或透過外部手段驗證（例如直接查詢資料庫而不是使用介面）。警告訊號：你重構時測試失敗，但行為沒有改變。如果你重命名一個內部函式而測試失敗，那些測試是在測試實作，而不是行為。

參見 [tests.md](tests.md) 了解範例，參見 [mocking.md](mocking.md) 了解模擬指引。

## 反模式：水平切片

**不要先寫所有測試，然後寫所有實作。** 這是「水平切片」——把 RED 視為「寫所有測試」，GREEN 視為「寫所有程式碼」。

這會產生**爛測試**：

- 批量寫的測試在測試*想像中的*行為，而不是*實際的*行為
- 你最終測試的是事物的*形狀*（資料結構、函式簽名），而不是面向使用者的行為
- 測試對真實變化不敏感——行為破壞時通過，行為正常時失敗
- 你超越了你的視野，在理解實作之前就提交了測試結構

**正確的方法**：透過曳光彈的垂直切片。一個測試 → 一個實作 → 重複。每個測試回應你從上一個循環學到的東西。因為你剛剛寫了程式碼，你確切知道什麼行為重要以及如何驗證它。

```
錯誤（水平）：
  RED:   test1, test2, test3, test4, test5
  GREEN: impl1, impl2, impl3, impl4, impl5

正確（垂直）：
  RED→GREEN: test1→impl1
  RED→GREEN: test2→impl2
  RED→GREEN: test3→impl3
  ...
```

## 工作流程

### 1. 規劃

探索程式碼庫時，讀取 `CONTEXT.md`（如果存在），讓測試名稱和介面詞彙符合專案的領域語言，並遵循你正在觸碰的區域的 ADR。

在寫任何程式碼之前：

- [ ] 與使用者確認需要哪些介面變更
- [ ] 與使用者確認要測試哪些行為（優先排序）
- [ ] 識別深層模組的機會（小介面，深實作）——執行 `/codebase-design` skill 取得詞彙和可測試性檢查
- [ ] 列出要測試的行為（不是實作步驟）
- [ ] 取得使用者對計畫的批准

問：「公開介面應該長什麼樣子？哪些行為最重要測試？」

**你無法測試所有東西。** 與使用者確認究竟哪些行為最重要。將測試精力集中在關鍵路徑和複雜邏輯上，而不是每個可能的邊界案例。

### 2. 曳光彈

寫一個測試，確認系統的一件事：

```
RED:   為第一個行為寫測試 → 測試失敗
GREEN: 寫最少程式碼使其通過 → 測試通過
```

這是你的曳光彈——證明路徑端對端可運作。

### 3. 增量迴路

對每個剩餘的行為：

```
RED:   寫下一個測試 → 失敗
GREEN: 最少程式碼使其通過 → 通過
```

規則：

- 每次一個測試
- 只有足以通過當前測試的程式碼
- 不要預測未來的測試
- 讓測試專注於可觀察的行為

### 4. 重構

所有測試通過後，尋找[重構候選項](refactoring.md)：

- [ ] 提取重複
- [ ] 加深模組（將複雜度移到簡單介面背後）
- [ ] 在自然的地方應用 SOLID 原則
- [ ] 思考新程式碼對現有程式碼揭示了什麼
- [ ] 每個重構步驟後執行測試

**在 RED 時絕不重構。** 先到達 GREEN。

## 每個循環的清單

```
[ ] 測試描述行為，而不是實作
[ ] 測試只使用公開介面
[ ] 測試在內部重構後仍然存活
[ ] 程式碼是這個測試的最少量
[ ] 沒有加入推測性功能
```
