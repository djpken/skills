Skill 的存在是為了從隨機系統中提取確定性。**可預測性**——agent 在每次執行中採取相同的*流程*，而不是產生相同的輸出——是根本美德；以下每個槓桿都服務於它。

**粗體術語**定義在 [`GLOSSARY.md`](GLOSSARY.md) 中；查閱那裡以獲得完整含義。

## 呼叫

兩個選擇，各有不同的代價：

- **模型呼叫（model-invoked）** skill 保留**描述（description）**，讓 agent 可以自主觸發它，*且*其他 skill 也能呼叫它（你仍然可以手動輸入其名稱）。它對**context 負載（context load）**有貢獻——描述每一個回合都坐在視窗中。機制：省略 `disable-model-invocation`，寫一個面向模型的描述，包含豐富的觸發措辭（「使用時機：使用者想要……，提到……」）。
- **使用者呼叫（user-invoked）** skill 從 agent 的觸及範圍中剝離描述：只有你輸入其名稱才能呼叫它——其他 skill 也無法。零 context 負載，但會花**認知負載（cognitive load）**：*你*是必須記得它存在的索引。機制：設定 `disable-model-invocation: true`；`description` 變成面向人類的——一行摘要，去掉觸發列表。

只有在 agent 必須自行呼叫 skill，或其他 skill 必須呼叫時，才選擇模型呼叫。如果它只有手動觸發，讓它成為使用者呼叫，不付 context 負載。

當使用者呼叫 skill 增多到你無法記住的程度，積累的認知負載可以用**路由器 skill（router skill）**來解決：一個使用者呼叫的 skill，列出其他 skill 以及何時應用每一個。

## 撰寫描述

模型呼叫的**描述**做兩件事——說明 skill 是什麼，並列出應該觸發它的**分支**。每個詞都增加**context 負載**，所以描述比正文更值得修剪：

- **前置 skill 的首要詞**——描述是它做呼叫工作的地方。
- **每個分支一個觸發。** 重新命名同一分支的同義詞是**重複（duplication）**——「使用 TDD 建構功能……要求測試先行開發」是一個分支寫了兩次。折疊它們；只保留真正不同的分支。
- **剪掉正文中已有的身份描述。** 保留觸發，加上任何「其他 skill 需要時……」的觸及條款。

## 資訊層次

Skill 由兩種內容類型組成——**步驟（steps）**和**參考（reference）**——自由混合：一個 skill 可以全是步驟、全是參考，或兩者兼有。核心決策是使用哪個，以及每個在**資訊層次**上的位置，按 agent 多快需要材料排名：

1. **Skill 內步驟** — `SKILL.md` 中的有序動作，主要層：agent 按順序做什麼。每個步驟以**完成標準（completion criterion）**結束，告訴 agent 工作已完成的條件。讓它*可檢查*（agent 能區分完成和未完成嗎？），以及在重要的地方*詳盡*（「每個修改過的模型都有說明」，而不是「產生變更列表」）——模糊的標準會導致**過早完成（premature completion）**。
2. **Skill 內參考** — `SKILL.md` 中的定義、規則或事實，按需查閱。通常是合理的扁平同等集（每條規則在同一層次）——這是好的安排，不是問題。*這個 skill 全是參考。*
3. **外部參考** — 從 `SKILL.md` 推出到獨立檔案中的參考，通過**context 指針（context pointer）**觸及，只在指針觸發時載入。（跨越*公開*參考——一個兄弟檔案如 `GLOSSARY.md`，仍然是 skill 的一部分——直到完全**外部參考**，存在於 skill 系統之外，任何 skill 都可以指向它。）

苛刻的完成標準驅動徹底的**腿力（legwork）**——agent 在工作中做的深挖——無論 skill 有步驟還是沒有，因為「每條規則都應用了」綁定扁平參考，就像「每個步驟完成了」綁定序列一樣。

推得太少，頂層就膨脹；推得太多，你就隱藏了 agent 實際需要的材料。這個張力就是整個決策。

**漸進披露（Progressive disclosure）**是沿著層次往下移動的動作——從 `SKILL.md` 到一個連結的檔案——讓頂層保持可讀。機制：在 skill 資料夾中有一個連結的 `.md` 檔案，以其所包含的內容命名（這個 skill 將完整定義披露給 `GLOSSARY.md`）。一些 skill 以多種方式使用，每種不同的方式是一個**分支**——在 skill 的不同執行中採取不同路徑。分支是最乾淨的披露測試：內嵌每個分支都需要的東西，並把只有某些分支才會觸及的東西推到指針後面。**context 指針**的*措辭*，而不是其目標，決定 agent 何時以及多可靠地觸及材料。

在層次決定某個部分坐在*多深*的地方，**共同定位（co-location）**決定一旦在那裡，*什麼坐在它旁邊*：將概念的定義、規則和警告放在一個標題下，而不是分散開，讓讀一部分帶來鄰近的內容。

## 何時拆分

**粒度（Granularity）**是你如何細分 skill，每次切割都花掉兩種負載之一，所以只在切割值得時才拆分。兩種切割：

- **按呼叫拆分** — 當你有一個不同的**首要詞（leading word）**應該自行觸發時，或另一個 skill 必須觸及它時，拆分出一個**模型呼叫**的 skill。你為新的始終載入的**描述**付**context 負載**，所以那個獨立觸及必須值得。
- **按序列拆分** — 當後面的**步驟**（一個步驟的**後完成步驟（post-completion steps）**）誘使 agent 倉促完成前面的步驟（**過早完成**）時，拆分一個**步驟**序列。把它們保持在視野之外鼓勵 agent 對當前任務做更多**腿力**。

## 修剪

將每個含義保持在**單一真相來源（single source of truth）**：一個權威的地方，所以改變行為是一個地方的編輯。

檢查每一行的**相關性**：它還與 skill 的工作有關嗎？

然後逐句而不只是逐行獵找**無操作（no-ops）**：對每個句子單獨執行無操作測試，當一個失敗時，刪除整個句子而不是修剪詞語。要積極——大多數失敗的散文應該消失，而不是重寫。

## 首要詞

**首要詞（leading word）**是一個緊湊的概念，已經存在於模型的預訓練中，agent 在執行 skill 時用它思考（例如，*lesson*、*fog of war*、*tracer bullets*）。在整個文本中反覆出現（雖然不一定——一個強大的首要詞可能只需要出現一次），它積累了分散的定義，並通過招募模型已持有的先驗，以最少的 token 錨定整個行為區域。

它服務兩次可預測性。在正文中，它錨定*執行*：每次詞語出現時，agent 都能呼叫相同的行為。在描述中，它錨定*呼叫*：當相同的詞語出現在你的提示、文件和程式碼中時，agent 將那個共享的語言連結到 skill 並更可靠地觸發它。

尋找機會將 skill 重構為使用首要詞。在三個地方拼出的三元組（**重複**），一個描述花一個句子指向一個想法——每個都是一段在乞求**折疊（collapse）**成一個 token 的文字。例子包括：

- 「快速、確定性、低開銷」-> *tight*——一個跨越一個階段的品質被重述——折疊成一個單一的預訓練詞（一個*緊密的*迴路）。
- 「你相信的迴路」-> *red*——將模糊的門轉換為二元可觀察的狀態（迴路在 bug 上變*紅*，或者不）。

你雙重獲益：更少的 token，*和*一個更清晰的鉤子讓 agent 掛上思考。假設每個 skill 都攜帶首要詞可以退休的重述——去找它們。

## 失敗模式

使用這些來診斷使用者可能在 skill 方面遇到的問題。

- **過早完成（Premature completion）** — 在步驟真正完成之前結束它，注意力滑向*完成*。防禦，按順序：首先磨練完成標準（便宜、局部）；只有當它不可化簡地模糊*且*你觀察到了倉促，才通過拆分隱藏後完成步驟（序列切割）。
- **重複（Duplication）** — 同樣的含義在多個地方。增加維護成本和 token，並在層次上將含義的重要性膨脹超過其實際等級。
- **沉積（Sediment）** — 因為添加感覺安全而移除感覺有風險而沉澱的過時層。任何沒有修剪紀律的 skill 的預設命運。
- **蔓延（Sprawl）** — 一個純粹太長的 skill，即使每一行都是活的和獨特的。損害可讀性和可維護性，並浪費 token。解決方案是層次：在指針後面披露**參考**，並按**分支**或序列拆分，讓每條路徑只攜帶它需要的東西。
- **無操作（No-op）** — 模型預設就已遵守的一行，所以你付負載卻什麼都沒說。測試：它與預設相比改變了行為嗎？弱的首要詞（當 agent 已經算是徹底的時候，*要徹底*）是無操作；修復方法是一個更強的詞（*無情*），而不是不同的技術。
