# 真正工程師的 Skills

[![skills.sh](https://skills.sh/b/mattpocock/skills)](https://skills.sh/mattpocock/skills)

> [English](./README.md) · 繁體中文 · [情境技能路由 →](./docs/scenario-routing.md)

我每天用來做真正工程的 Agent 技能——不是在 vibe coding。

開發真實應用程式很難。GSD、BMAD、Spec-Kit 等方法試圖透過掌控整個流程來幫助你，但這樣反而讓你失去控制，流程裡的 bug 也很難追查。

這些 skills 設計得小巧、易於調整，並且可以組合使用。它們適用於任何模型，基於數十年的工程經驗。你可以自由改造，讓它們成為你自己的工具。

如果你想追蹤這些 skills 的更新，歡迎訂閱電子報，已有約 60,000 位開發者加入：

[訂閱電子報](https://www.aihero.dev/s/skills-newsletter)

## 快速開始（30 秒設定）

1. 執行 skills.sh 安裝工具：

```bash
npx skills@latest add mattpocock/skills
```

2. 選擇你要的 skills，以及想安裝到哪些 coding agent 上。**記得選取 `/setup-matt-pocock-skills`**。

3. 在你的 agent 中執行 `/setup-matt-pocock-skills`，它會詢問：
   - 你使用哪個 issue 追蹤工具（GitHub、Linear 或本地檔案）
   - 你在 triage 票據時套用的標籤名稱（`/triage` 會用到）
   - 你想把產出的文件存在哪裡

4. 搞定，可以開始用了。

## 為什麼需要這些 Skills

我建立這些 skills，是為了解決我在 Claude Code、Codex 和其他 coding agent 上觀察到的常見失敗模式。

### 失敗模式 #1：Agent 沒做我要的東西

> 「沒有人確切知道自己想要什麼。」
>
> David Thomas & Andrew Hunt，[《The Pragmatic Programmer》](https://www.amazon.co.uk/Pragmatic-Programmer-Anniversary-Journey-Mastery/dp/B0833F1T3V)

**問題**：軟體開發中最常見的失敗模式是**對齊失敗**。你以為 dev 知道你要什麼，結果看到成品才發現完全不是那回事。

AI 時代也一樣。你和 agent 之間存在溝通落差。解法是一場**深度訪談（grilling session）**——讓 agent 主動問你一堆詳細問題。

**解法**：

- [`/grill-me`](./skills/productivity/grill-me/SKILL.md)——適用於非程式碼場景
- [`/grill-with-docs`](./skills/engineering/grill-with-docs/SKILL.md)——同上，但附加更多功能（見下方）

這是我最受歡迎的兩個 skills。它們幫助你在動手前先和 agent 對齊，並深入思考你要做的改動。**每次**要做改動前都應該用一次。

### 失敗模式 #2：Agent 說話太囉唆

> 在通用語言（ubiquitous language）下，開發者的對話和程式碼的表達都從同一個領域模型衍生出來。
>
> Eric Evans，[《Domain-Driven Design》](https://www.amazon.co.uk/Domain-Driven-Design-Tackling-Complexity-Software/dp/0321125215)

**問題**：一開始，開發者和領域專家通常說的是不同語言。Agent 被丟進一個專案，要自己摸索術語，結果用 20 個字說一個字就能講清楚的事。

**解法**：建立一套共同語言——一份幫助 agent 解碼專案術語的文件。

<details>
<summary>範例</summary>

以下是 `course-video-manager` 的 [`CONTEXT.md`](https://github.com/mattpocock/course-video-manager/blob/076a5a7a182db0fe1e62971dd7a68bcadf010f1c/CONTEXT.md) 範例。哪一句比較容易讀？

- **之前**：「在課程某個章節裡的課程被『實體化』（即在檔案系統中分配位置）時出現問題」
- **之後**：「materialization cascade 有問題」

這種精確性在每次 session 都能持續節省 token。

</details>

這個功能內建在 [`/grill-with-docs`](./skills/engineering/grill-with-docs/SKILL.md) 裡。它是一個 grilling session，同時幫你和 AI 建立共同語言，並以 ADR 格式記錄難以解釋的決策。

> [!TIP]
> 共同語言除了減少囉嗦之外，還有許多好處：
>
> - **變數、函式和檔案命名一致**，使用共同語言
> - **程式碼庫更容易被 agent 導覽**
> - Agent **思考時消耗更少 token**，因為有更精準的語言可用

### 失敗模式 #3：程式碼就是跑不起來

> 「永遠採取小而謹慎的步驟。反饋速率就是你的速度上限。永遠不要接下太大的任務。」
>
> David Thomas & Andrew Hunt，[《The Pragmatic Programmer》](https://www.amazon.co.uk/Pragmatic-Programmer-Anniversary-Journey-Mastery/dp/B0833F1T3V)

**問題**：假設你和 agent 已對齊要建什麼，但 agent 生出來的東西還是爛的。問題在於反饋迴路——沒有對程式碼實際執行情況的回饋，agent 就是在瞎飛。

**解法**：你需要完整的反饋迴路：靜態型別、瀏覽器存取、自動化測試。

對於自動化測試，紅綠重構迴路（red-green-refactor）至關重要。讓 agent 先寫一個失敗的測試，再修到通過。這能給 agent 一致的反饋，大幅提升程式碼品質。

我建了 **[`/tdd`](./skills/engineering/tdd/SKILL.md) skill**，可插入任何專案使用。它鼓勵紅綠重構，並給 agent 充分的測試好壞指引。

對於 debug，我也建了 **[`/diagnose`](./skills/engineering/diagnose/SKILL.md)** skill，將最佳 debug 實踐包裝成一個簡單的迴路。

### 失敗模式 #4：我們蓋出了一堆爛泥巴

> 「每天都要投資系統的設計。」
>
> Kent Beck，[《Extreme Programming Explained》](https://www.amazon.co.uk/Extreme-Programming-Explained-Embrace-Change/dp/0321278658)

> 「最好的模組是深層的。它們讓大量功能可以透過簡單的介面存取。」
>
> John Ousterhout，[《A Philosophy Of Software Design》](https://www.amazon.co.uk/Philosophy-Software-Design-2nd/dp/173210221X)

**問題**：大多數用 agent 建出的應用程式都很複雜、難以修改。因為 agent 大幅加速了寫程式的速度，同時也加速了軟體熵的增長。程式碼庫以前所未有的速度變得越來越複雜。

**解法**：一個全新的 AI 開發方法：**在意程式碼的設計**。

這個理念貫穿這些 skills 的每一層：

- [`/to-prd`](./skills/engineering/to-prd/SKILL.md) 在建立 PRD 之前，會詢問你要動哪些模組
- [`/zoom-out`](./skills/engineering/zoom-out/SKILL.md) 要求 agent 在整個系統脈絡下解釋程式碼

最重要的是，[`/improve-codebase-architecture`](./skills/engineering/improve-codebase-architecture/SKILL.md) 幫助你拯救已經變成爛泥巴的程式碼庫。我建議每隔幾天執行一次。

### 總結

軟體工程的基本功比以往任何時候都更重要。這些 skills 是我將這些基本功濃縮成可重複實踐的最佳嘗試，幫助你在職業生涯中交出最好的作品。享受它們。

## 技能總覽

### Engineering（工程）

我每天寫程式時使用的 skills。

- **[diagnose](./skills/engineering/diagnose/SKILL.md)** — 針對難搞 bug 和效能退化的嚴謹診斷迴路：重現 → 最小化 → 假設 → 測量 → 修復 → 回歸測試。
- **[grill-with-docs](./skills/engineering/grill-with-docs/SKILL.md)** — 深度訪談 session，挑戰你的計畫是否符合現有領域模型，精煉術語，並同步更新 `CONTEXT.md` 和 ADR。
- **[triage](./skills/engineering/triage/SKILL.md)** — 透過狀態機角色流程，對 issue 進行分類。
- **[improve-codebase-architecture](./skills/engineering/improve-codebase-architecture/SKILL.md)** — 根據 `CONTEXT.md` 的領域語言和 `docs/adr/` 的決策記錄，找出程式碼庫的深化機會。
- **[setup-matt-pocock-skills](./skills/engineering/setup-matt-pocock-skills/SKILL.md)** — 建立每個 repo 的設定（issue 追蹤工具、triage 標籤詞彙、領域文件格局），供其他 engineering skills 使用。在使用 `to-issues`、`to-prd`、`triage`、`diagnose`、`tdd`、`improve-codebase-architecture` 或 `zoom-out` 之前，每個 repo 執行一次。
- **[tdd](./skills/engineering/tdd/SKILL.md)** — 測試驅動開發，採紅綠重構迴路。每次一個垂直切片地實作功能或修 bug。
- **[to-issues](./skills/engineering/to-issues/SKILL.md)** — 將任何計畫、spec 或 PRD 拆解成可獨立認領的 GitHub issues（垂直切片）。
- **[to-prd](./skills/engineering/to-prd/SKILL.md)** — 將目前對話脈絡轉換成 PRD，並以 GitHub issue 提交。不做訪談——直接綜合你已討論的內容。
- **[zoom-out](./skills/engineering/zoom-out/SKILL.md)** — 要求 agent 拉遠視角，在整個系統脈絡下解釋陌生的程式碼區段。
- **[prototype](./skills/engineering/prototype/SKILL.md)** — 快速建立可拋棄的原型，釐清設計方向——針對狀態／商業邏輯問題建立可執行的 terminal app，或是從同一個 route 切換的多個截然不同的 UI 版本。

### Productivity（生產力）

通用工作流程工具，不限於寫程式。

- **[caveman](./skills/productivity/caveman/SKILL.md)** — 超壓縮溝通模式。去掉廢話，token 用量減少約 75%，同時保持完整技術精準度。
- **[grill-me](./skills/productivity/grill-me/SKILL.md)** — 針對計畫或設計進行無情訪談，直到每個決策分支都解決為止。
- **[handoff](./skills/productivity/handoff/SKILL.md)** — 將目前對話壓縮成交接文件，讓另一個 agent 可以接續工作。
- **[teach](./skills/productivity/teach/SKILL.md)** — 在多個 session 中教導使用者新技能或概念，以當前目錄作為有狀態的教學工作區。
- **[write-a-skill](./skills/productivity/write-a-skill/SKILL.md)** — 用正確結構、漸進式揭露和附帶資源，建立新的 skills。

### Misc（雜項）

偶爾用到的工具。

- **[git-guardrails-claude-code](./skills/misc/git-guardrails-claude-code/SKILL.md)** — 設定 Claude Code hooks，在危險 git 指令（push、reset --hard、clean 等）執行前攔截。
- **[migrate-to-shoehorn](./skills/misc/migrate-to-shoehorn/SKILL.md)** — 將測試檔案從 `as` 型別斷言遷移至 @total-typescript/shoehorn。
- **[scaffold-exercises](./skills/misc/scaffold-exercises/SKILL.md)** — 建立包含章節、題目、解答和解說的練習目錄結構。
- **[setup-pre-commit](./skills/misc/setup-pre-commit/SKILL.md)** — 設定 Husky pre-commit hooks，整合 lint-staged、Prettier、型別檢查和測試。
